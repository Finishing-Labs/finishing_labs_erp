System Architecture

Finishing Labs ERP follows a classic MVC (Model-View-Controller) architecture pattern with modern enhancements.

Technology Stack

  • Backend Framework: Flask (Python)
  • Database: SQLAlchemy ORM with SQLite/PostgreSQL
  • Frontend: HTML5, CSS3, HTMX
  • Template Engine: Jinja2

Design Principles

  • Component-Based: Reusable template components (sidebar, topbar)
  • HTMX-First: Minimizes JavaScript, server-driven interactions
  • RESTful Routes: Clean URL structure and HTTP verb usage
  • Single Responsibility: Each controller manages one domain

Setup Instructions

Prerequisites

  • Python 3.8 or higher
  • pip (Python package manager)
  • Virtual environment tool (venv or virtualenv)

Installation

# Clone the repository
git clone <repository-url>
cd finishing_labs_erp

# Create virtual environment
python -m venv venv

# Activate virtual environment
# Windows:
.\venv\Scripts\activate
# Mac/Linux:
source venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Initialize the database
flask db upgrade

# Run the application
python app.py
# or use the start script:
.\start.ps1

Configuration

Configuration is managed in config.py. Key settings include:

  • SECRET_KEY: Session encryption key
  • SQLALCHEMY_DATABASE_URI: Database connection string
  • DEBUG: Development mode toggle

Database Schema

The system uses SQLAlchemy ORM for database interactions. Models are defined in the models/ directory.

Core Entities

  • Customer: Stores customer information and contact details
  • Order: Represents customer orders with items and status
  • PackingSlip: Generated shipping documentation
  • ProductionItem: Tracks items through production stages

Relationships

  • Customer → Orders (one-to-many)
  • Order → PackingSlips (one-to-many)
  • Order → ProductionItems (one-to-many)

Controllers

Controllers handle HTTP requests and coordinate between models and views.

Controller Structure

Each controller is a Flask Blueprint registered in app.py:

from flask import Blueprint, render_template, request

controller_name = Blueprint('name', __name__)

@controller_name.route('/path')
def index():
    # Handle request
    return render_template('template.html')

Available Controllers

  • dashboard.py: Dashboard and analytics
  • customers.py: CRUD operations for customers
  • orders.py: Order creation and management
  • packing_slips.py: Packing slip generation
  • production.py: Production workflow and builder

Template System

Template Architecture

Templates follow a hierarchical inheritance pattern:

  • base_layout.html: Root template with HTML structure
  • Module index.html: Full page with sidebar/topbar
  • Module content.html: HTMX-targeted content only

Component Pattern

Shared UI components (sidebar, topbar) are extracted into templates/components/ for reusability.

{% extends "base_layout.html" %}

{% block content %}
    {% include "components/sidebar.html" %}
    {% include "components/topbar.html" %}
    
    <main>
        {# Page content #}
    </main>
{% endblock %}

HTMX Integration

HTMX enables dynamic interactions without writing JavaScript. See HTMX_REFERENCE.md for details.

Common Patterns

  • hx-get: Load content dynamically
  • hx-post: Submit forms via AJAX
  • hx-target: Specify where to insert response
  • hx-swap: Control how content is swapped

Example

<button hx-get="/orders/new" 
        hx-target="#content" 
        hx-swap="innerHTML">
    New Order
</button>

Extending the System

Adding a New Module

  1. Create a new controller in controllers/module_name.py
  2. Define routes using Flask Blueprint
  3. Create model(s) in models/ if needed
  4. Add templates in templates/module_name/
  5. Register the blueprint in app.py
  6. Add navigation link to sidebar component

Best Practices

  • Follow the existing naming conventions
  • Create both full page and content-only templates
  • Use HTMX for dynamic interactions when possible
  • Keep controllers thin, move business logic to models/services
  • Write database migrations for schema changes
  • Update documentation when adding features

API Endpoints

When adding new endpoints, follow RESTful conventions:

  • GET /resource - List all
  • GET /resource/new - New form
  • POST /resource - Create
  • GET /resource/:id - View one
  • GET /resource/:id/edit - Edit form
  • PUT /resource/:id - Update
  • DELETE /resource/:id - Delete

Testing

Guidelines for testing the application:

  • Use pytest for unit and integration tests
  • Test controllers with Flask test client
  • Mock database interactions where appropriate
  • Test HTMX endpoints return correct partial HTML

Deployment

Recommended deployment options:

  • Production Server: Gunicorn or uWSGI
  • Reverse Proxy: Nginx or Apache
  • Database: PostgreSQL for production
  • Platforms: Heroku, Railway, DigitalOcean, AWS
